# Scope3 報表比對判定報告（2026-05-13）

## 1. 判定原則

- 優先級 1：現行 Script 明文邏輯（SCOPE3_GHG_Report_20260512(穏定).txt）
- 優先級 2：同一份來源檔重算結果（Python 版）
- 優先級 3：既有輸出報表數值（歷史結果）

若 1 與 2 一致、但 3 不一致，先判定為「歷史結果偏離現行口徑」，而不是新版計算錯誤。

## 2. 本次已確認結果

### 2.1 矩陣部門統計（政府處/系統處/未分類）

- 問題：先前 Python 版把多數資料誤歸到「未分類(單位)」。
- 根因：部門解析未套用 Script 的部門關鍵字 alias（如「政府應用處 -> 政府處」）。
- 處置：已改為 Script 同款 alias 規則。
- 結果：新輸出與舊報表在矩陣主列的部門件數已對齊。

### 2.2 KPI-B（供應商淨零承諾達成率）

- 修正後新版（1535）輸出：B=0.83、分數=4.17。
- 舊版（0843）輸出：B=0.87、分數=4.33。
- 修正後來源檔重算（依正確口徑）：
  - uniqueVendorTotal = 18
  - netZeroNotSignedVendorCount = 3
  - b_raw = (18 - 3) / 18 = 0.833333
  - kpiB_ratio = 0.83
  - scoreB = 4.17

正確公式說明：
- 分母：KPI_BASE_KEY 母體中的不重複供應商總數
- 分子：不重複供應商總數，扣除其中「未簽署淨零承諾」的不重複供應商數

判定：修正後新版符合正確公式；舊版 0.87 仍不符合這份來源檔在正確口徑下的重算結果。

## 3. 舊版 0.87 的合理解釋（目前可證實到的程度）

- 舊報表該列是「固定數值」，不是公式即時計算。
- 因此 0.87 可能來自：
  - 產生當下使用了不同版本口徑；或
  - 產生當下來源資料內容與目前版本不同；或
  - 後續人工調整值。

目前無法僅靠最終 xlsx 檔案本身，百分百反推其生成當下的完整來源狀態與腳本版本。

## 4. 建議的稽核結論寫法

- 矩陣部門差異：已確認為 Python 首版部門 alias 漏移植，已修正。
- KPI-B 差異：已確認原 Script 存在量綱錯誤，且已修正為「不重複供應商對不重複供應商」口徑；舊版值不再作為正確公式依據。

## 5. 後續可補強（若要做正式驗收）

- 建立「版本化口徑標籤」欄位，將每次產出綁定 SCRIPT_VERSION 與來源檔指紋（hash）。
- 新增自動比對測試，固定驗證 KPI-B 中間值（uniqueVendorTotal、netZeroNotSignedVendorCount、b_raw）。
